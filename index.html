  • 涉及技术：SpringBoot、MyBatis、Guava Cache、Redis、Dubbo、ZooKeeper、RocketMQ 等

  • 项目简述：一个基于 Java 技术栈进行开发的 Web 项目，遵循微服务设计架构思想，把书店项目拆分为商品模块、用户模块、订单模块、优惠券模块，模块间通过 RPC 框架进行远程服务的调用。在系统优化方面，加入了缓存来减缓数据库压力加快响应速度，使用消息队列对并发下单场景进行流量削峰

  • 项目效果
    1、实现了用户注册、登录，商品展示、收藏、下单、撤销订单、结账等基本功能
    2、增加了优惠券的查询、领取、使用功能
    3、在公告板匿名展示最近用户优惠券使用情况
    4、管理员可以查询、取消订单，新增、删除、修改商品、优惠券的信息

  • 开发细节
    1、模块间使用 Dubbo 实现服务之间的远程调用，并使用 ZooKeeper 作为服务的注册中心，实现服务自动注册、发现
    2、商品、优惠券以及公告板信息三者属于读多写少且公开的信息，故可以加载到缓存中供其他接口查询，这样可以减轻数据库压力并减少查询时间。商品、优惠券的信息需要定时回写到数据库更新旧记录的值并从数据库加载新记录，很适合使用 loading cache 实现；公告板信息只由订单模块产生且数据量少无需保存到数据库，可以使用 Redis 存储并持久化到磁盘
    3、订单模块和优惠券模块使用 RocketMQ 通信，既可以达到订单模块与优惠券模块解耦的效果，又可以实现并发下单支付场景下的流量削峰。优惠券模块对于下单和支付两种不同的信息作出不同的数据更新操作
    4、考虑到在分布式场景下使用 UUID 可能出现订单号或券码不唯一的情况，故采用雪花算法生成唯一订单号或券码

  • 遇到的问题及解决方案
    1、使用 RocketMQ 发送消息超时抛出异常导致部分消息丢失，最终查看源代码发现默认超时时间为 3s ，显式设置为 30s 后问题得以解决
    2、订单模块重复消费 RocketMQ 中的消息，断点调试后发现是因为无法正确把消息体中的数据反序列化为对象导致无法正确返回消费成功的标记，最终解决方法为使用阿里开源 JSON 处理工具 FastJSON 进行序列化与反序列化
    3、本地缓存 loading cache 的自动刷新机制只能刷新已有记录为最新值，当数据库中插入了新的记录，新的记录无法加载到本地缓存，解决方案为使用监听器配合 Spring 定时任务每隔五分钟手动把数据库最新的数据加载到本地缓存
    4、在优惠券模块消费消息的过程中会偶尔收到一些不明来历的信息，通过断点调试查看这些消息的格式发现与订单模块发送的消息格式不一致，所以不会反序列化成功也不会消费成功，只会一直重复消费。网上搜索资料发现可能是低版本 RocketMQ 本身的 BUG，由于这些消息不会影响正常消息的收发，所以最终处理方法是消费前先过滤出这些无用信息并直接返回已消费标记

  • 项目收获：学习到一个项目是如何进行整体需求分析、模块拆分的，理解不同功能模块间配合工作的过程，加深了对于开发过程中规范的制定和实行的理解。在实现基本功能的基础上应该去思考系统潜在的问题，比如高并发场景下是否可用、高效。也会去思考系统可以优化的地方以及如何进行技术选型，比如本地缓存与分布式缓存的权衡
